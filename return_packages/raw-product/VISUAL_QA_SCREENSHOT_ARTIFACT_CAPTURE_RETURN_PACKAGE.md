# Visual QA Screenshot Artifact Capture Return Package

## What Changed

- Added a Playwright-backed Node artifact runner for the existing 20-case
  `/visual-qa` surface:
  `frontend/tools/captureVisualQaScreenshots.mjs`.
- Added an explicit frontend browser dependency and npm command for screenshot
  artifacts.
- Added a thin top-level Makefile entry point.
- Updated UI and Visual QA planning docs with the artifact command, output
  contract, prerequisites, and the current no-diff boundary.

The runner keeps the existing runtime boundaries:

- It targets the current `/visual-qa` page and existing
  `data-visual-case-id="<case_id>"` card selectors.
- It does not change the Visual QA corpus, case definitions, product rendering,
  backend behavior, parser/routing behavior, or result contracts.
- It writes generated artifacts under ignored `outputs/` paths. It does not add
  screenshot diffing, committed PNG baselines, or a CI gate.

## Command Surfaces Added

Frontend command:

```bash
npm --prefix frontend run qa:visual-screenshots -- \
  --base-url http://127.0.0.1:8000 \
  --run-id <run_id>
```

Top-level Makefile command:

```bash
make visual-qa-screenshots \
  VISUAL_QA_BASE_URL=http://127.0.0.1:8000 \
  VISUAL_QA_RUN_ID=<run_id>
```

Targeted reruns can repeat `--case <case_id>` on the npm command. The default
remains the canonical all-case run.

## Artifact Contract Implemented

The runner writes:

```text
outputs/visual_qa_screenshots/<run_id>/
  manifest.json
  desktop_1280/
    metrics.json
    page.png
    cards/<case_id>.png
  mobile_390/
    metrics.json
    page.png
    cards/<case_id>.png
```

Canonical capture scope:

- desktop viewport: `desktop_1280`, `1280x900`
- mobile viewport: `mobile_390`, `390x844`
- one full-page PNG per viewport
- one card PNG for every case in the 20-case Visual QA page

Metrics include:

- viewport label, target dimensions, and measured inner dimensions
- live run-progress text and parsed completed/total counts
- loaded case count
- request error count
- backend `ok`, `noResult`, and `error` status counts
- document and body client/scroll widths
- selected card count and captured case IDs
- observed card count and observed page case IDs
- overflow pass/fail detail

The command fails for:

- `/visual-qa` load or completion failure
- request errors
- loaded-response gaps
- missing selected case IDs
- duplicate or empty page case IDs
- canonical case-card count drift from 20
- measured document/body horizontal overflow
- screenshot capture failures
