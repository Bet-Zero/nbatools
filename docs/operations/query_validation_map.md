# Query Validation Map

## Purpose

NBA Tools has several query-validation layers. They answer different
questions and must not be cited as if they were interchangeable.

The Raw QA harness is implemented by `tools/raw_query_answer_qa.py`. Its
curated case source is `qa/raw_query_answer_corpus.yaml`. Saved subset
selectors live under `qa/harness_slices/`. Python query tests are configured
separately through `Makefile` and `pyproject.toml`.

Exploratory query review is implemented separately by
`tools/exploratory_query_review.py`. Its default input is
`qa/exploratory_query_samples.yaml`. It runs input-only samples and writes
human-inspection snapshots; it is not expectation coverage.

## Validation Layers

### Full Raw QA Corpus

- Source: `qa/raw_query_answer_corpus.yaml`
- Command:

  ```bash
  .venv/bin/python tools/raw_query_answer_qa.py \
    --corpus qa/raw_query_answer_corpus.yaml \
    --fail-on-expectation-failure
  ```

- Meaning: runs all corpus cases from the YAML.
- Important: this is the full corpus, not a slice and not pytest.

### Raw QA Corpus Slice

- Source: `qa/harness_slices/<slice>.yaml`, selecting case IDs from the full
  corpus.
- Command:

  ```bash
  .venv/bin/python tools/raw_query_answer_qa.py \
    --corpus qa/raw_query_answer_corpus.yaml \
    --slice <slice> \
    --fail-on-expectation-failure
  ```

- Meaning: runs a named subset of the full corpus.
- Important: a slice is not a separate corpus and a slice result is not a
  full-corpus result.

### Product Review Doc

- Output: `outputs/raw_query_answer_qa/<run_id>/product_review.md`
- Meaning: human-readable review artifact generated by the Raw QA harness.
- Important: this is not a separate corpus and not a separate test source.

The product review doc helps a reviewer inspect family coverage,
representative answers, suspicious rows, and product decisions. It does not
change the scope of the harness run that generated it.

### Exploratory Query Review

- Source: `qa/exploratory_query_samples.yaml` or another input-only YAML/JSON
  sample file.
- Command:

  ```bash
  .venv/bin/python tools/exploratory_query_review.py \
    --input qa/exploratory_query_samples.yaml
  ```

- Output: `outputs/exploratory_query_review/<run_id>/`
- Meaning: human-readable and JSONL snapshots of what the shared natural-query
  engine returned for each sample.
- Important: this is not Raw QA, not pytest, and not validation evidence. A
  reviewed exploratory row becomes regression coverage only after it is
  manually promoted into `qa/raw_query_answer_corpus.yaml` with explicit
  expectations.

### Rendered UI Review

- Output: `outputs/public_ui_render_review/<run_id>/`
- Meaning: browser-rendered screenshot/text review for selected representative
  queries.
- Important: this checks actual UI presentation, not the whole corpus unless
  explicitly run that way.

### Pytest Query Tests

- Command: `make test-query`
- Expands to: `pytest -m query`
- Meaning: Python tests marked `query`.
- Important: this is separate from the Raw QA corpus and should not be cited
  as corpus evidence.

## How To Talk About Results

Use these terms precisely:

- `machine-passing`: Raw QA expectations passed for the stated corpus scope:
  full corpus, slice, or single case.
- `product-reviewed`: a human/ChatGPT reviewed the generated
  `product_review.md` or equivalent answer artifact.
- `UI-reviewed`: rendered browser output/screenshots/text snapshots were
  reviewed.
- `deferred polish`: known issue, not blocking the current validation scope.

Reporting rules:

- Do not say "the corpus passed" unless the full unsliced Raw QA corpus was
  run.
- Say "the `public_query_acceptance` slice passed" when only that slice was
  run.
- Do not mix pytest query counts with corpus counts.
- Do not treat return packages as durable source-of-truth docs.

| `qa/raw_query_answer_corpus.yaml` | full curated Raw QA corpus |
| `qa/exploratory_query_samples.yaml` | default exploratory review sample input |
| `qa/harness_slices/` | named Raw QA subset selectors |
| `outputs/exploratory_query_review/` | generated exploratory review snapshots; not validation evidence |
| `outputs/raw_query_answer_qa/` | generated Raw QA reports and product-review artifacts |
| `outputs/public_ui_render_review/` | generated rendered UI review artifacts |

Return packages may serve as historical receipts for a particular handoff.
They are not authoritative validation docs and are not long-term evidence for
the current scoreboard.
