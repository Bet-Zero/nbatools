# Core Result/Table Contracts

This reference locks the current Wave 1 and Wave 2 route -> result shape -> table pattern
contracts used by the React result renderer. It documents current behavior only:
no visual redesign, no route-registry redesign, and no new query capability.

Source files:

- `frontend/src/components/results/config/routeToPattern.ts`
- `frontend/src/components/results/resultShapes.ts`
- `frontend/src/components/results/patterns/*Result.tsx`
- `src/nbatools/commands/structured_results.py`

## Route-Specific Leaderboard Supporting Columns

These rows lock current frontend display behavior for generic `LeaderboardResult`
routes that now have route-aware column configs. This is a frontend display
contract only; backend result shapes did not change.

| Route/family | Locked visible columns | Hidden fields | Future backend enrichment ideas |
|---|---|---|---|
| `season_leaders` | `#`, `Player`, primary metric, `Team`, `GP`, shooting makes/attempts when present (`FGM`, `FGA`, `3PM`, `3PA`, `FTM`, `FTA`), `Season`, `Season Type`. | `player_id`; any unconfigured row keys not in the locked support list. | MPG and richer secondary stat context for season leader rows. |
| `season_team_leaders` | `#`, `Team`, primary metric, optional `W-L`/`Win %` record context when wins/losses/win percentage are present, `GP`, shooting makes/attempts when present, `Season`, `Season Type`. `opponent_pts_per_game` is the primary metric for points-allowed leaderboards. | `team_id`; unrelated row keys such as broad team profile stats unless they are the primary metric or locked context. | Team PPG/net profile context for record-like team leader rows. |
| `team_record_leaderboard` | `#`, `Team`, `W-L`, `Win %`, `Games`, `Season` or `Seasons`, `Season Type`. If the selected metric is wins or losses, `W-L` is highlighted as the primary record metric. | `team_id`; separate raw wins/losses columns are not shown because `W-L` carries the record display. | Team PPG, Opp PPG, margin/net context if backend emits trustworthy record leaderboard support stats. |
| `player_occurrence_leaders` | `#`, `Player`, primary occurrence count, `Team`, `GP`, `Season`, `Season Type`. | `player_id`; no rate/frequency columns are invented. | Occurrence rate/frequency and row-level event descriptors. |
| `team_occurrence_leaders` | `#`, `Team`, primary occurrence count, `GP`, `Season`, `Season Type`. Team abbreviation renders cleanly as identity when no team name is present. | `team_id`; no rate/frequency columns are invented. | Occurrence rate/frequency and cleaner event labels supplied by backend metadata. |
| `playoff_appearances` ranked variant | `#`, `Team`, `Appearances`, `Round`, `Seasons` or `Season`. | Team IDs and unrelated postseason metadata. | First/last playoff appearance fields. |
| `lineup_leaderboard` | `#`, `Lineup`, `Team`, primary metric defaulting to `Net`, `Minutes`, `ORtg`, `DRtg`, `Pace`, `TS%`. Lineup labels use `lineup_name`, `player_names`, or legacy synthetic lineup keys. | `lineup_id`, `player_ids`, raw pipe-delimited `player_names` when rendered as the lineup label, `unit_size`, `minute_minimum`, `season`, `season_type`. | Backend metadata-driven lineup column choice and a dedicated lineup leaderboard pattern if the UI needs richer lineup detail. |

Specialized `TopPerformancesResult` routes remain unchanged in this wave.
Player top-game score fields remain a future backend enrichment idea because
player top-game rows do not reliably expose team/opponent scores today.

QA/data-quality policy for extreme top-performance rows: official-source-backed
outliers are displayed as-is, not capped or suppressed. The raw answer QA
harness flags extreme values for review and records verified official outliers
in `qa/verified_outliers.yaml`; future high-point rows that are not allowlisted
remain suspicious review flags. Hard impossibilities remain validation failures
or review blockers.

## Current Weak Contracts

- Configured generic leaderboard routes now use route-specific supporting columns. Unknown or unconfigured leaderboard payloads still infer supporting columns from row keys and display-order rules.
- Game-log product row cap is frontend-level (`12` rows) while backend finder limits are route-specific. This is current behavior -- needs product decision if pagination or row caps should change.
- Detail drawers are opt-in by pattern config and only appear when undisplayed row fields exist. This is current behavior -- needs product decision if detail drawers should remain.
- `fallback_table` remains available for unmatched routes, but Wave 1 routes are explicitly guarded from fallback use.
- Wave 2 routes are explicitly guarded from fallback use. `lineup_summary` required a narrow production mapping to leave fallback behavior.
- `player_on_off` currently renders a split pattern over `SummaryResult.summary` rows keyed by `presence_state`; this is accepted current behavior for now, and it does not use the `SplitSummaryResult`/`split_comparison` section convention.
- `record_by_decade` stores decade rows in the `by_season` section for backward compatibility.
- `playoff_appearances` has two frontend-supported variants under the same backend route: ranked `leaderboard` results render as a leaderboard, and single-team `summary` plus `by_season` results render through playoff-history appearances mode.
- `lineup_summary` now renders through the generic entity-summary pattern with lineup-aware hero text, but it does not yet have a dedicated lineup table or illustration.
- `routeToPattern` remains a route-specific switch statement. This wave intentionally did not convert it to a registry.
